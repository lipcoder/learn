这一节 7.1 的核心其实就一句话：

**接口不是“这个值是什么”，而是“这个值能做什么”。** ([books.studygolang.com][1])

---

## 1. 先理解“具体类型”和“接口类型”的区别

这一节先对比了两类类型：

* **具体类型**：比如 `int`、`[]string`、`bytes.Buffer`。你知道它内部是什么，也知道它支持哪些操作。([books.studygolang.com][1])
* **接口类型**：它不告诉你内部结构，只告诉你“这个值至少有这些方法可以调用”。也就是一种**行为约定**。([books.studygolang.com][1])

你可以把它理解成：

* 具体类型：像“某品牌某型号的打印机”
* 接口类型：像“只要能打印的设备”

程序在依赖接口时，不关心你到底是激光打印机、喷墨打印机，还是虚拟 PDF 打印机，只关心你**会不会打印**。

---

## 2. 这节为什么用 `fmt.Fprintf` 举例

书里说，`fmt.Printf` 输出到标准输出，`fmt.Sprintf` 返回字符串，但它们都复用了 `fmt.Fprintf`。`Printf` 传入的是 `os.Stdout`，`Sprintf` 则创建了一个 `bytes.Buffer`，然后把内容写进去。([books.studygolang.com][1])

页面里的代码是这个思路：

```go
func Fprintf(w io.Writer, format string, args ...interface{}) (int, error)

func Printf(format string, args ...interface{}) (int, error) {
    return Fprintf(os.Stdout, format, args...)
}

func Sprintf(format string, args ...interface{}) string {
    var buf bytes.Buffer
    Fprintf(&buf, format, args...)
    return buf.String()
}
```

这里最关键的点是：

**`Fprintf` 的第一个参数不是“文件”，而是 `io.Writer` 接口。** ([books.studygolang.com][1])

也就是说，`Fprintf` 不在乎目标到底是：

* 终端文件 `*os.File`
* 内存缓冲区 `*bytes.Buffer`
* 还是别的什么对象

它只要求：**你得有 `Write([]byte) (int, error)` 这个方法。** ([books.studygolang.com][1])

---

## 3. `io.Writer` 到底是什么

页面给出的定义是：

```go
type Writer interface {
    Write(p []byte) (n int, err error)
}
```

也就是：凡是实现了这个 `Write` 方法的类型，都可以被当成 `io.Writer` 使用。([books.studygolang.com][1])

所以这一节想表达的是：

> `Fprintf` 和调用者之间，不是通过“某个具体类型”合作，而是通过“接口约定”合作。([books.studygolang.com][1])

这就是“接口是合约”的意思。

### 你可以这么理解“合约”

`Fprintf` 说：

> 你别告诉我你是什么类型，我不关心。
> 你只要承诺：我给你一串字节时，你能用 `Write` 接住。

调用者说：

> 好，我虽然不是文件，但我实现了 `Write`，所以我也能接。

---

## 4. 这有什么好处

最大的好处是：**解耦**。

`Fprintf` 不需要分别写很多版本：

* `FprintfToFile`
* `FprintfToBuffer`
* `FprintfToSocket`
* `FprintfToHttpResponse`

它只写一个版本，面向 `io.Writer` 就够了。([books.studygolang.com][1])

这带来两个直接收益：

### （1）复用

同一套格式化逻辑，可以写到不同地方。`Printf` 和 `Sprintf` 就是现成例子。([books.studygolang.com][1])

### （2）可替换性

这一节提到，只要两个具体类型都满足同一个接口，就可以相互替换，这就是**可替换性**（文中提到 LSP，里氏替换原则）。([books.studygolang.com][1])

通俗说：

* 今天传 `*os.File`
* 明天传 `*bytes.Buffer`
* 后天传你自己写的类型

`Fprintf` 都能工作。

## 5. `ByteCounter` 这个例子为什么很重要

书里写了一个新类型：

```go
type ByteCounter int

func (c *ByteCounter) Write(p []byte) (int, error) {
    *c += ByteCounter(len(p))
    return len(p), nil
}
```

这个类型并不真的保存写入内容，它只是**统计一共“写入”了多少字节**，然后把这个长度累加起来。([books.studygolang.com][1])

因为它实现了 `Write`，所以它也满足 `io.Writer`，于是就可以直接传给 `fmt.Fprintf`。([books.studygolang.com][1])

书里的效果是：

```go
var c ByteCounter
c.Write([]byte("hello"))
fmt.Println(c) // 5

c = 0
var name = "Dolly"
fmt.Fprintf(&c, "hello, %s", name)
fmt.Println(c) // 12
```

为什么最后是 12？

因为 `"hello, Dolly"` 一共 12 个字节。`Fprintf` 在格式化时把结果交给 `c.Write`，`ByteCounter` 只负责计数。([books.studygolang.com][1])

### 这个例子真正想说明什么

它想说明：

**一个类型只要实现了接口要求的方法，就能接入现有通用代码。**

哪怕这个类型的“真实用途”很特别：

* 文件：把字节写到磁盘
* 缓冲区：把字节存到内存
* `ByteCounter`：根本不存，只计数

对 `Fprintf` 来说都一样，因为它只认 `io.Writer`。([books.studygolang.com][1])

---

## 6. 这里为什么 `Write` 是指针接收者

`ByteCounter` 的方法是：

```go
func (c *ByteCounter) Write(...)
```

不是：

```go
func (c ByteCounter) Write(...)
```

原因很简单：

它要修改计数器本身的值。如果不用指针接收者，改的是副本，外面的 `c` 不会变。这个结论也和页面代码展示出的累加行为一致。([books.studygolang.com][1])

## 7. 这一节还引出了另一个重要接口：`fmt.Stringer`

页面后半段说，除了 `io.Writer`，`fmt` 包还很依赖另一个接口：

```go
type Stringer interface {
    String() string
}
```

只要一个类型实现了 `String() string`，它在被 `fmt.Print` 等输出时，就可以按你定义的字符串形式来展示。([books.studygolang.com][1])

比如书里提到：

* `Celsius` 类型可以打印成 `"100°C"`
* `*IntSet` 可以打印成集合形式 `"{1 2 3}"` ([books.studygolang.com][1])

这说明接口不只用来“写入”，也可以用来“定制输出表现”。

## 8. 你学完这一节，脑子里要留下的 4 句话

### 第一，接口描述行为，不描述数据结构

接口关心“能做什么”，不是“里面长什么样”。([books.studygolang.com][1])

### 第二，面向接口编程能减少重复

`Printf`、`Sprintf` 复用 `Fprintf`，就是因为它们都能把目标抽象成 `io.Writer`。([books.studygolang.com][1])

### 第三，实现接口不需要显式声明

这一页虽然没展开讲“implements”语法，但从 `ByteCounter` 被直接传给 `Fprintf` 可以看出：只要方法集满足接口要求，就能用。([books.studygolang.com][1])

### 第四，接口让替换变得自然

文件、缓冲区、计数器都能接到同一个 `Fprintf` 上，这就是接口带来的扩展性。([books.studygolang.com][1])

## 9. 用一句很接地气的话总结 7.1

这一节其实是在教你：

> **写函数时，尽量依赖“需要的能力”，而不是依赖“具体的类型”。**

比如：

* 你需要“可写”能力，就收 `io.Writer`
* 你需要“可读”能力，就收 `io.Reader`
* 你需要“能转成字符串”，就用 `fmt.Stringer`

这样你的代码会更通用、更容易复用、更容易扩展。这个思想正是 7.1 的核心。([books.studygolang.com][1])

# 1. 到底是谁“提供这个方法”？

看这个接口：

```go
type Writer interface {
    Write(p []byte) (n int, err error)
}
```

它的意思不是：

* `io.Writer` 自己会帮你写 `Write`
* 也不是 `Fprintf` 里面去实现 `Write`

而是：

* **谁想被当成 `io.Writer` 来用，谁就自己提供 `Write` 方法**

也就是：

* `*os.File` 自己实现 `Write`
* `*bytes.Buffer` 自己实现 `Write`
* 你写的 `*ByteCounter` 也自己实现 `Write`

---

# 2. 接口本身是什么角色？

接口本身更像一个“招聘要求”或者“插头标准”。

比如：

```go
type Writer interface {
    Write([]byte) (int, error)
}
```

它只是说：

> 想来应聘 `Writer` 这个岗位，可以。
> 但你必须会 `Write` 这个方法。

它**只提要求，不提供实现**。

---

# 3. `Fprintf` 在干嘛？

`Fprintf` 的定义大概是：

```go
func Fprintf(w io.Writer, format string, args ...interface{}) (int, error)
```

它的意思是：

> 你给我一个 `w`，这个 `w` 不一定是文件，也不一定是缓冲区。
> 但它必须满足 `io.Writer`，也就是它必须会 `Write`。

然后 `Fprintf` 自己负责：

1. 先把 `"hello, %s"` 和参数 `"Tom"` 拼成结果
2. 变成字节
3. 调用 `w.Write(...)` 把内容交出去

所以：

* **格式化字符串** 是 `Fprintf` 做的
* **真正写出去** 是 `w` 做的

---

# 4. 谁调用谁？

关系是这样的：

* `Printf` 调 `Fprintf`
* `Sprintf` 也调 `Fprintf`
* `Fprintf` 再调传进来的那个对象的 `Write`

所以真正的调用链是：

```text
Printf/Sprintf
    ↓
  Fprintf
    ↓
w.Write(...)
```

# 5. 先看最简单的理解

## 情况 A：`Printf`

```go
fmt.Printf("hello %s", "Tom")
```

大致等价于：

```go
fmt.Fprintf(os.Stdout, "hello %s", "Tom")
```

这里传进去的是 `os.Stdout`。

* `os.Stdout` 是标准输出
* 它对应的类型支持 `Write`
* 所以它满足 `io.Writer`

于是流程是：

1. `Printf` 偷偷调用 `Fprintf(os.Stdout, ...)`
2. `Fprintf` 格式化出 `"hello Tom"`
3. `Fprintf` 调 `os.Stdout.Write(...)`
4. 内容显示到终端

---

## 情况 B：`Sprintf`

```go
s := fmt.Sprintf("hello %s", "Tom")
```

大致等价于：

```go
var buf bytes.Buffer
fmt.Fprintf(&buf, "hello %s", "Tom")
s := buf.String()
```

这里传进去的是 `&buf`。

* `bytes.Buffer` 也实现了 `Write`
* 所以它也满足 `io.Writer`

于是流程是：

1. `Sprintf` 创建一个缓冲区 `buf`
2. 调 `Fprintf(&buf, ...)`
3. `Fprintf` 格式化出 `"hello Tom"`
4. `Fprintf` 调 `buf.Write(...)`
5. 内容写进内存，不是终端
6. 最后 `buf.String()` 取出字符串

# 7. 你可以把它想成“快递站”

* `Fprintf` = 打包员
* `io.Writer` = 收货方标准
* `Write` = 收货动作

`Fprintf` 说：

> 我只负责把东西打包好。
> 至于送给谁，我不管。
> 但对方必须有“收货能力”，也就是必须有 `Write` 方法。

所以：

* `os.Stdout`：我能收，收到后显示到屏幕
* `bytes.Buffer`：我能收，收到后放内存里
* `ByteCounter`：我也能收，但我不保存，我只数一下多少字节

# 8. `ByteCounter` 为什么也能传进去？

因为它自己提供了 `Write`：

```go
type ByteCounter int

func (c *ByteCounter) Write(p []byte) (int, error) {
    *c += ByteCounter(len(p))
    return len(p), nil
}
```

所以它也符合 `io.Writer` 的要求。

注意：

* `io.Writer` 没有“点名”说只能文件才能实现
* 任何类型，只要有这个方法，都行

这就是 Go 接口最厉害的地方：

> **不看出身，只看能力。**

# 9. 一句最关键的话

你现在要牢牢记住这句：

> **接口只定义“需要什么方法”，具体方法实现由具体类型自己提供。**

所以这里分工是：

* `io.Writer`：提要求
* `Fprintf`：使用这个要求
* `os.Stdout` / `bytes.Buffer` / `ByteCounter`：真正实现这个要求

# 10. 最后用伪代码把关系捋一遍

## 接口定义要求

```go
type Writer interface {
    Write([]byte) (int, error)
}
```

## 一个具体类型自己实现

```go
type MyBuffer struct {}

func (m *MyBuffer) Write(p []byte) (int, error) {
    // 真正处理数据
    return len(p), nil
}
```

## Fprintf 使用接口

```go
func Fprintf(w io.Writer, format string, args ...interface{}) {
    data := 格式化后的字符串
    w.Write([]byte(data))
}
```

## 调用

```go
var b MyBuffer
Fprintf(&b, "hello %s", "Tom")
```

这里：

* `Fprintf` 不认识 `MyBuffer`
* 但它看到 `&b` 有 `Write`
* 所以就能用

2. **输出阶段**

```go
}
```

也就是说，`Printf` 自己其实只是个“简单封装”，它把“输出目标”固定成了 `os.Stdout`。([books.studygolang.com][1])

## 你问“哪里需要接口”

答案是：

**不是 `Printf` 这个名字本身需要接口，而是 `Fprintf` 为了通用性，需要把“输出目标”抽象成接口。**
因为如果没有接口，`Fprintf` 就只能写死成“往文件写”或者“往标准输出写”。有了 `io.Writer`，它就能接收任何实现了 `Write` 的东西，比如 `*os.File` 和 `*bytes.Buffer`。([books.studygolang.com][1])

所以：

* `Printf`：固定写到终端
* `Sprintf`：固定写到内存缓冲区
* `Fprintf`：不固定写到哪里，只要求“你给我的目标会 `Write`”

这正是书里说的：`Printf` 和 `Sprintf` 都是对 `Fprintf` 的封装。([books.studygolang.com][1])

## 只盯着 `Printf` 的最小流程图

```text
你写：
fmt.Printf("hello %s", "Tom")

实际大致变成：
fmt.Fprintf(os.Stdout, "hello %s", "Tom")

然后 Fprintf 做两件事：
1. 把格式串和参数整理成 "hello Tom"
2. 调用 os.Stdout.Write([]byte("hello Tom"))

最后：
终端显示 hello Tom
```

这里最关键的一行是：

```text
os.Stdout.Write(...)
```

因为 `Fprintf` 不关心 `os.Stdout` 到底是什么具体类型，它只关心：

```text
你有没有 Write 方法？
```

而这正是 `io.Writer` 这个接口在约束的事。书里明确说了，`io.Writer` 定义了 `Fprintf` 和调用者之间的约定：调用者提供一个具有 `Write` 方法的具体值，而 `Fprintf` 保证对任何满足这个接口的值都能工作。([books.studygolang.com][1])

* `Printf` 只是把“写到哪儿”固定成 `os.Stdout`
* 真正体现接口价值的是 `Fprintf`，因为它把“写到哪儿”抽象掉了。([books.studygolang.com][1])

## 先直接回答你的判断

你说：

> “相当于是 `Fprintf` 在检查第一个参数有没有这个 `Write` 这个方法啊”

**可以这么理解，但要稍微修正一下：**

不是说 `Fprintf` 自己在代码里手动 `if` 一下去检查。
而是：

* `Fprintf` 把第一个参数类型写成了 `io.Writer`
* 所以**调用 `Fprintf` 的时候，Go 就要求你传进去的值必须满足这个接口**
* 满足的标准就是：这个值所属的具体类型，得有 `Write([]byte) (int, error)` 这个方法。书里明确说，`Fprintf` 和调用者之间的约定，就是调用者提供像 `*os.File`、`*bytes.Buffer` 这样带有特定签名 `Write` 方法的具体值，而 `Fprintf` 保证对任何满足 `io.Writer` 的值都能工作。([Go语言书籍][1])

所以更准确地说：

> **不是 `Fprintf` 在“手动检查”，而是 `Fprintf` 通过参数类型把要求写死了：你给我的东西必须是 `io.Writer`。**

## `io.Writer` 到底体现在哪里？

就体现在这句函数签名里：

```go
func Fprintf(w io.Writer, format string, args ...interface{}) (int, error)
```

这里的 `w io.Writer` 就是在说：

> `w` 可以是很多不同的具体类型，
> 但它们都必须满足 `io.Writer` 这个接口约定。([Go语言书籍][1])

所以，`io.Writer` 不是某段会运行的“额外代码”，它更像是：

* 一个**参数要求**
* 一个**能力说明**
* 一个**调用约束**

## 为什么最后看到的是 `os.Stdout.Write(...)`？

因为你在 `Printf` 这条链路里，传进去的具体值就是 `os.Stdout`。

书里的代码是：

```go
func Printf(format string, args ...interface{}) (int, error) {
    return Fprintf(os.Stdout, format, args...)
}
```

也就是说：

* `Printf` 调 `Fprintf`
* 传进去的 `w` 恰好是 `os.Stdout`
* 而 `os.Stdout` 的具体类型是 `*os.File`
* `*os.File` 有 `Write` 方法，所以它满足 `io.Writer`。([Go语言书籍][1])

于是到了真正执行写入的时候，效果上当然就会落到：

```text
os.Stdout.Write(...)
```

这并不和接口矛盾。
**接口只是把“要求”抽象出来，真正干活的永远还是某个具体类型的方法。**

## 你现在可以把它理解成“两层”

### 第一层：抽象层

`Fprintf` 只认：

```text
我需要一个 io.Writer
```

也就是：

```text
我需要一个会 Write 的对象
```

### 第二层：具体层

这次 `Printf` 给它的是：

```text
os.Stdout
```

所以最后实际调用的就是：

```text
os.Stdout.Write(...)
```

## 一个很贴切的类比

你可以把 `io.Writer` 想成“门禁要求”：

> 进这个门的人，必须带工牌。

`Fprintf` 说：

> 我这里只收“带工牌的人”（也就是 `io.Writer`）。

而 `os.Stdout` 来了以后：

> 它正好带了工牌，因为它有 `Write` 方法。

所以最后真正进去干活的是 `os.Stdout`，不是“工牌”本身。

也就是说：

* `io.Writer` = 入场标准
* `os.Stdout` = 实际来干活的人
* `Write` = 它具备的能力

## 所以接口不是“多余的一层”吗？

不是。
因为如果没有 `io.Writer`，`Fprintf` 就只能写成类似这样：

```go
func FprintfToStdout(f *os.File, ...)
```

那它就只能给文件用。
但现在它写成 `io.Writer` 后，书里举的例子里它不仅能接 `*os.File`，还能接 `*bytes.Buffer`，甚至还能接 `*ByteCounter`，因为这些具体类型都满足 `Write` 这个约定。([Go语言书籍][1])

所以接口的价值是：

> **让 `Fprintf` 不用关心“你是谁”，只关心“你会不会 `Write`”。** ([Go语言书籍][1])

这个问题特别好。
你现在问的是这段里的：

```go
func Printf(format string, args ...interface{}) (int, error) {
    return Fprintf(os.Stdout, format, args...)
}
```

这里其实有 **两个点**：

1. `interface{}` 是什么意思
2. `...` 是什么意思

把这两个拆开就清楚了。

---

## 1. `interface{}` 是什么意思

`interface{}` 叫做**空接口**。

意思是：

> **它不要求任何方法。**

你前面看到的 `io.Writer`：

```go
type Writer interface {
    Write(p []byte) (n int, err error)
}
```

它要求必须有 `Write` 方法。

但 `interface{}` 是：

```go
interface{}
```

里面什么都没有，所以它的意思就是：

> 只要是任意类型的值，都可以放进来。

所以这些都可以是 `interface{}`：

* `123`
* `"hello"`
* `3.14`
* `true`
* 一个结构体
* 一个指针

---

## 2. `args ...interface{}` 是什么意思

这个整体的意思是：

> **`args` 是一个可变参数，里面可以放 0 个或多个任意类型的值。**

比如：

```go
fmt.Printf("%d %s %v\n", 18, "abc", true)
```

这里 `format` 是：

```go
"%d %s %v\n"
```

而 `args` 就相当于接收了后面的这些值：

```go
18, "abc", true
```

因为这些值类型不同：

* `18` 是 `int`
* `"abc"` 是 `string`
* `true` 是 `bool`

所以就不能写成：

```go
args ...int
```

因为那样只能接收很多个 `int`。

`fmt.Printf` 要支持各种类型，所以它写成：

```go
args ...interface{}
```

意思就是：

> 后面你爱传什么类型都行，我都收。

---

## 3. 这个 `...` 到底是什么意思

在**参数定义里**：

```go
args ...interface{}
```

意思是：

> 这是一个**可变参数**。

也就是调用时可以写：

```go
Printf("%d", 1)
Printf("%d %d", 1, 2)
Printf("%s %d %v", "tom", 18, true)
Printf("hello")
```

都可以。

---

## 4. 在函数内部，`args` 实际上是什么

虽然定义的时候写的是：

```go
args ...interface{}
```

但在函数里面，你可以把它理解成：

> **`args` 实际上就是一个 `[]interface{}` 切片。**

也就是说，到了函数内部，它大概像这样：

```go
[]interface{}{18, "abc", true}
```

所以你可以把它想成：

```go
func Printf(format string, args []interface{})
```

只是调用时 `...interface{}` 更方便，允许你直接传很多个参数。

---

## 5. 那为什么下面又写 `args...`

这句特别关键：

```go
return Fprintf(os.Stdout, format, args...)
```

这里的 `args...` 和前面的定义不是一回事。

这里是在**调用函数**时使用 `...`。

意思是：

> 把 `args` 这个切片“拆开”，一个一个传给 `Fprintf`。

---

### 你要区分这两个位置

### （1）定义参数时

```go
args ...interface{}
```

意思是：

> 这个函数可以接收多个任意类型参数

---

### （2）调用函数时

```go
args...
```

意思是：

> 把一个切片展开成多个参数传进去

---

## 6. 为什么这里必须写 `args...`

因为 `Fprintf` 的定义也是：

```go
func Fprintf(w io.Writer, format string, args ...interface{}) (int, error)
```

它也要的是“多个参数”。

而 `Printf` 里面的 `args`，在函数体内其实是一个切片：

```go
[]interface{}
```

所以如果你想把它继续原样转交给 `Fprintf`，就要写：

```go
args...
```

意思是把这个切片展开。

---

## 7. 一个非常直观的小例子

### 先定义函数

```go
func test(args ...interface{}) {
    fmt.Println(args)
}
```

### 调用 1：直接传多个值

```go
test(1, "abc", true)
```

效果上相当于：

```go
args = []interface{}{1, "abc", true}
```

---

### 调用 2：已经有一个切片

```go
a := []interface{}{1, "abc", true}
test(a...)
```

这里的 `a...` 就是把切片展开成：

```go
test(1, "abc", true)
```

---

### 如果你不写 `...` 会怎样？

比如：

```go
a := []interface{}{1, "abc", true}
test(a)   // 注意：这里没有 ...
```

这时候传进去的是：

> 一个参数，这个参数本身是 `[]interface{}`

不是三个参数。

这个差别很重要。

---

## 8. 回到 `Printf` 这段代码

```go
func Printf(format string, args ...interface{}) (int, error) {
    return Fprintf(os.Stdout, format, args...)
}
```

它的意思其实就是：

1. `Printf` 收到格式串和很多个参数
2. 这些参数类型可以不同，所以用 `...interface{}`
3. `Printf` 自己不做特殊事
4. 它只是把这些参数**原封不动转交**给 `Fprintf`
5. 但因为 `args` 在函数体内是一个切片，所以转交时要写 `args...`

---

## 9. 你可以把它翻译成人话

这段代码相当于说：

> `Printf`：
> 你把格式串和一堆任意类型的值给我。
> 我再把标准输出 `os.Stdout` 补进去，
> 然后把这堆参数转手交给 `Fprintf`。

也就是：

```text
Printf(format, 参数们...)
    ->
Fprintf(os.Stdout, format, 参数们...)
```

---

## 10. 最后给你一个最短总结

### `interface{}`

表示：

> 任意类型

### `args ...interface{}`

表示：

> 0 个或多个任意类型参数

### `args...`

表示：

> 把 `args` 这个切片展开，继续当作可变参数传出去

---

你现在可以把整句先记成：

```go
func Printf(format string, args ...interface{})
```

就是：

> `Printf` 接收一个格式字符串，外加若干个任意类型的值。

而这句：

```go
Fprintf(os.Stdout, format, args...)
```

就是：

> 把这些值继续传给 `Fprintf`，并且输出目标固定成 `os.Stdout`。
